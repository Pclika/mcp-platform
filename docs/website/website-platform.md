# Website Platform Plan

## Goal

Build a website that is not just a store, but a guided platform experience for product selection, documentation, and developer onboarding.

## Website Roles

The website should do four jobs:

- explain the platform
- help users choose the right kit and modules
- provide documentation and examples
- convert technical users into product buyers or B2B leads

## Main Sections

- Home
- MCP Ready Kit
- Modules
- Solutions
- Docs
- Examples
- GitHub
- B2B Services
- Lead Time and Policy

## Key User Journeys

### 1. New Developer

Needs:

- understand what the platform is
- choose a starter kit
- read quickstart docs
- see examples

### 2. AI Tool User

Needs:

- import the GitHub repository
- understand project structure
- pick a capability path
- connect the right hardware

### 3. B2B Team

Needs:

- understand platform extensibility
- evaluate industrial options
- request adaptation or small-batch support

## Guided Configuration Flow

The website should include a simple chooser with a few options.

Suggested options:

- What are you building: sensing, vision, motion, control, communication
- Who are you: maker, educator, startup, industrial team
- What do you need first: starter kit, expansion pack, service

Output:

- recommended kit
- recommended modules
- recommended examples
- recommended docs

## Front-End Priorities

- strong technical clarity
- clean product hierarchy
- visible GitHub and docs entry
- easy comparison between Basic, Pro, and Industrial
- module capability map
- clear lead time messaging for the no-finished-goods model

## Why This Matters

The website is part of the platform.

It should reduce configuration friction and make the open-source foundation directly useful for real hardware adoption.
